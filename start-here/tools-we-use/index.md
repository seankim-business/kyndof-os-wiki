# 우리가 쓰는 도구들

킨도프에서 쓰는 주요 도구들과 그 이유를 설명합니다.

**처음부터 다 알 필요 없습니다.** 필요할 때 찾아보세요.

---

## 커뮤니케이션

### Slack

**용도:** 실시간 대화, 공지, 협업

**주요 채널:**
- `#general` - 전체 공지 및 자유 대화
- `#it-test` - 기술 질문, 봇 테스트
- `#questions` - 뭐든 물어보는 곳
- `#random` - 업무 외 잡담
- 팀별 채널 (예: `#brand-ops`, `#atelier`)

**사용 팁:**
- 급한 건 DM 대신 채널에 (다른 사람도 배울 수 있음)
- 스레드 기능 활용 (대화 정리)
- 이모지 반응으로 간단히 응답
- 방해 금지 시간 설정 (퇴근 후)

**에티켓:**
- 긴급하지 않으면 @channel 사용 자제
- "안녕하세요"만 보내지 말고, 질문까지 한 번에
- 답변 받으면 감사 표시 (👍 이모지라도)

### 이메일

**용도:** 공식 커뮤니케이션, 외부 연락

**언제 쓰나:**
- 외부 파트너/고객과 소통
- 공식 기록이 필요한 경우
- 긴 문서나 첨부파일 공유

**언제 안 쓰나:**
- 내부 팀원과 빠른 대화 → Slack
- 프로젝트 협업 → Notion
- 코드 리뷰 → GitHub

## 문서 & 협업

### Notion

**용도:** 모든 업무 문서, 프로젝트 관리, 지식 베이스

**주요 공간:**
- **Company** - 회사 전체 문서 (비전, 전략, OKR)
- **Teams** - 팀별 워크스페이스
- **Projects** - 진행 중인 프로젝트들
- **Knowledge** - 가이드, 매뉴얼, SOP

**자주 쓰는 기능:**
- 데이터베이스 (프로젝트, 태스크 관리)
- 템플릿 (회의록, 프로젝트 브리프)
- 페이지 링크 (`@` 멘션)
- 체크리스트

**사용 팁:**
- 페이지는 깊게 만들기 (계층 구조 활용)
- 템플릿 적극 활용
- 중요한 건 즐겨찾기에 추가
- 검색 단축키 익히기 (Cmd+K)

### GitHub

**용도:** 코드 저장소, 버전 관리, 협업 (개발자/기술직)

**주요 레포지토리:**
- `kyndof-os` - CompanyOS 시스템
- `runtime` - 에이전트 런타임
- `wiki` - 이 Wiki (내부 문서)
- 팀별 프로젝트 레포

**기본 워크플로우:**
1. 이슈 생성 (작업 내용 정의)
2. 브랜치 생성 (`feature/기능명`)
3. 코드 작성 및 커밋
4. PR (Pull Request) 생성
5. 리뷰 후 머지

**사용 팁:**
- 커밋 메시지는 명확하게 (무엇을, 왜)
- PR은 작게 자주 (큰 PR은 리뷰하기 힘듦)
- 리뷰는 빠르게, 친절하게
- 이슈에 컨텍스트 충분히 적기

## 디자인 & 크리에이티브

### Figma

**용도:** UI/UX 디자인, 프로토타입, 디자인 시스템

**주요 파일:**
- Design System - 킨도프 디자인 시스템
- 프로젝트별 디자인 파일
- 와이어프레임 & 프로토타입

**사용 팁:**
- 컴포넌트 재사용 (일관성)
- 버전 히스토리 활용
- 코멘트로 피드백
- 개발자와 협업 (인스펙트 모드)

## 개발 & 배포

### Railway

**용도:** 서버 배포, 호스팅

**주요 서비스:**
- Production - 실제 운영 환경
- Staging - 테스트 환경

**자주 쓰는 기능:**
- 로그 확인 (에러 디버깅)
- 환경 변수 설정
- 배포 롤백 (문제 발생 시)

## 도구 선택 원칙

킨도프는 도구를 신중하게 선택합니다.

### 왜 이 도구들인가?

**Slack vs 카카오톡:**
- 업무/개인 분리
- 검색 가능한 대화 히스토리
- 봇 연동 (자동화)

**Notion vs Google Docs:**
- 데이터베이스 기능 (프로젝트 관리)
- 위키 스타일 구조 (지식 축적)
- 템플릿 시스템 (반복 작업 효율화)

**GitHub vs 다른 버전관리:**
- 오픈소스 생태계
- 협업 기능 (PR, 리뷰)
- CI/CD 연동

### 새 도구를 도입할 때

1. **문제 정의**: 현재 도구로 안 되는 게 뭔가?
2. **대안 검토**: 기존 도구로 해결 불가능한가?
3. **팀 논의**: 다른 팀원 의견 수렴
4. **시범 적용**: 작은 범위에서 테스트
5. **평가**: 실제로 문제를 해결했는가?

**도구는 수단입니다. 목적이 아닙니다.**

## 자주 묻는 질문

### "도구가 너무 많아요."

맞습니다. 처음엔 부담스럽습니다.

**첫 주에 집중할 것:**
- Slack (대화)
- Notion (문서 읽기)
- 이메일 (기본)

나머지는 필요할 때 배우세요.

### "어떤 도구를 언제 써야 하나요?"

**빠른 가이드:**

| 상황 | 도구 |
|------|------|
| 팀원에게 빠른 질문 | Slack |
| 프로젝트 문서 작성 | Notion |
| 외부 파트너 연락 | 이메일 |
| 코드 리뷰 | GitHub |
| 디자인 피드백 | Figma |
| 서버 로그 확인 | Railway |

### "다른 회사 도구 경험이 없어요."

괜찮습니다. 다들 처음엔 그랬습니다.

**배우는 방법:**
1. 일단 써보기 (실수해도 괜찮음)
2. 막히면 팀원에게 물어보기
3. 도구 가이드 문서 읽기 (Notion 내)
4. 유튜브 튜토리얼 (기본 사용법)

**2주면 익숙해집니다.**

### "계정이 너무 많아요. 비밀번호 관리는?"

**권장 방법:**

1. **비밀번호 관리자 사용**
   - 1Password, Bitwarden 등
   - 안전하고 편리함

2. **2FA (2단계 인증) 설정**
   - GitHub, Notion 등 중요 계정
   - 보안 강화

3. **회사 이메일을 기본으로**
   - 모든 계정의 기본 이메일로 사용
   - 나중에 찾기 쉬움
