# Day 3: 첫 번째 일을 시작하다

{: .highlight }
> 오늘부터 실제로 일을 시작합니다. 작게 시작해서, 프로세스를 익히는 것이 목표입니다.

이틀 동안 회사와 팀을 파악했습니다. 오늘은 **손을 움직이는 날**입니다. 완벽하지 않아도 됩니다. 배우는 과정입니다.

---

## 오늘의 목표

- [ ] 첫 번째 task 착수
- [ ] 팀 프로세스 한 바퀴 경험
- [ ] 막히면 도움 요청하기
- [ ] 팀 미팅 참여 (있다면)

---

## 오전: 첫 번째 Task 시작

### Task 이해하기

task를 받았다면, 시작하기 전에 명확히 이해합니다.

**확인해야 할 것:**
- **무엇을:** 구체적으로 뭘 만들거나 수정해야 하는지
- **왜:** 이 task가 왜 필요한지 (맥락)
- **어디에:** 어떤 파일, 어떤 시스템에서 작업하는지
- **언제까지:** 예상 완료 시점

**모르겠으면 바로 물어보기:**
```
"[task] 시작하려고 하는데,
[X] 부분이 잘 이해가 안 가서요.
[구체적 질문]이 맞을까요?"
```

### 개발자의 첫 Task

**보통 첫 task:**
- 오타 수정, 문구 변경
- 간단한 버그 수정
- 작은 기능 추가
- 테스트 케이스 작성

**진행 순서:**

1. **이슈/task 확인** (Notion, GitHub Issues, Linear 등)
2. **로컬 환경 셋업** (아직 안 했다면)
3. **브랜치 생성** (`feature/기능명` 또는 `fix/버그명`)
4. **코드 작성**
5. **로컬에서 테스트**
6. **커밋 & 푸시**
7. **PR(Pull Request) 생성**
8. **리뷰 요청**

### 비개발자의 첫 Task

**보통 첫 task:**
- 문서 초안 작성
- 데이터 정리
- 리서치 및 요약
- 간단한 분석

**진행 순서:**

1. **task 명확화** (Notion에서 요구사항 확인)
2. **관련 자료 수집**
3. **초안 작성**
4. **팀원/매니저에게 피드백 요청**
5. **수정 및 완료**

## 프로세스 경험하기

### 진행 상황 공유

**시작할 때:**
```
Slack 또는 task 코멘트:
"[task] 시작합니다!"
```

**중간에 (필요시):**
```
"[task] 진행 중입니다.
[진행 상황] 완료했고,
[다음 단계] 작업 중입니다."
```

**막혔을 때:**
```
"[task] 진행 중에 [문제] 때문에 막혔습니다.
[시도한 것]은 해봤는데 안 됐고,
도움 부탁드려도 될까요?"
```

### 도움 요청하기

**혼자 20분 넘게 막혀 있지 마세요.**

도움 요청은 부끄러운 게 아닙니다. 오히려:
- 팀 시간을 아껴줍니다
- 배움의 기회가 됩니다
- 소통 능력을 보여줍니다

**좋은 도움 요청:**
```
"[상황 설명]하려고 하는데,
[시도한 것 1]과 [시도한 것 2]를 해봤는데 안 됐습니다.
[구체적 질문] 도움 주실 수 있을까요?"
```

**나쁜 도움 요청:**
```
"안 돼요." (뭐가 안 되는지 모름)
"도와주세요." (뭘 도와줘야 하는지 모름)
```

### 피드백 받기

첫 번째 결과물에는 반드시 피드백이 있을 겁니다.

**피드백을 받을 때:**
- 방어적으로 반응하지 않기
- 질문으로 이해도 확인하기
- 메모해두기
- 감사 표현하기

**피드백 반영 후:**
```
"[피드백] 반영해서 수정했습니다.
확인 부탁드려요!"
```

## 오후: 팀 미팅 참여

### 미팅 종류

팀마다 다르지만, 보통:

| 미팅 | 빈도 | 내용 |
|------|------|------|
| 스탠드업 | 매일 | 어제 한 일, 오늘 할 일, 막힌 것 |
| 스프린트 플래닝 | 주간 | 이번 주 할 일 결정 |
| 회고 | 주간/격주 | 잘한 점, 개선할 점 |
| 1:1 | 주간 | 매니저와 개인 대화 |

### 스탠드업 참여

**첫 스탠드업이라면:**

```
"안녕하세요, [이름]입니다.
어제는 1:1 미팅과 문서 읽기 했고,
오늘은 [첫 task] 진행하고 있습니다.
막힌 건 아직 없습니다!"
```

**진행 중이라면:**
```
"[task] 진행 중입니다.
[X] 부분은 완료했고,
[Y] 작업 중입니다.
막힌 건 없습니다."
```

### 미팅에서 발언하기

첫 주에 미팅에서 한 번은 발언해보세요.

**부담 없는 발언:**
- 간단한 질문
- 공감 표현 ("저도 그 부분 궁금했어요")
- 짧은 의견 ("좋은 것 같습니다")

처음부터 길게 말할 필요 없습니다.

## Day 3 FAQ

### "Task가 너무 어려워요."

**괜찮습니다.** 첫 task가 어려울 수 있습니다.

1. 어디가 어려운지 구체화하기
2. 20분 이상 막히면 도움 요청
3. 페어(pair)로 함께 작업 요청해도 OK

### "너무 느린 것 같아요."

**첫 task는 원래 오래 걸립니다.**

- 환경 익히는 데 시간 필요
- 컨텍스트 파악하는 데 시간 필요
- 프로세스 배우는 데 시간 필요

**속도보다 제대로 하는 게 중요합니다.**

### "실수했어요."

**축하합니다!** 배우고 있다는 뜻입니다.

1. 당황하지 않기
2. 상황 파악하기
3. 필요하면 바로 알리기
4. 복구 방법 찾기
5. 다음에 같은 실수 안 하도록 메모

**같은 실수만 반복하지 않으면 됩니다.**

### "PR 리뷰가 무서워요."

**리뷰는 공격이 아니라 학습입니다.**

- 리뷰어는 코드를 더 좋게 만들려는 것
- 개인에 대한 평가가 아님
- 질문으로 대화하기

```
"이 부분 [X]로 한 이유가 있는데,
[Y]가 더 나을까요?"
```
